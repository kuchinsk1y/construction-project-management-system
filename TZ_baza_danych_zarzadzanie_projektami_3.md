# Технічне завдання: База даних системи управління будівельними проектами

**Призначення документа:** технічне завдання для розробки ERD-діаграми та детальної схеми таблиць PostgreSQL для модуля управління будівельними проектами.

**СУБД:** PostgreSQL (версія 15+)

---

## 1. Контекст бізнесу

**Компанія:** польська компанія, яка займається будівництвом промислових фотовольтаїчних ферм (PV). У найближчому майбутньому планує будівництво промислових накопичувачів енергії (BESS). У подальшому можуть з'являтися інші типи будівельних проектів.

**Географія:** Польща та країни ЄС. Оплати зазвичай у валюті контракту.

**Мета системи:** замінити попередню систему на Google Sheets (покривала лише PV-проекти) на повноцінний веб-додаток з універсальною структурою, яка підтримує різні типи будівельних проектів.

**Технічний стек:**
- Бекенд + фронтенд — окремий веб-додаток
- БД — PostgreSQL
- Google Sheets — опціонально, для експорту даних керівництву (звичка)

---

## 2. Ролі в системі

| Роль | Повноваження |
|---|---|
| **Операційний директор** (Dyrektor Operacyjny) | Створює проекти, призначає менеджерів, визначає milestones, планує бюджет, розподіляє години на типи робіт |
| **Фінансовий директор** (Dyrektor Finansowy) | Планує доходи/витрати, виставляє та відстежує фактури, контролює платежі, аналізує рентабельність |
| **Менеджер проекту** (Kierownik Projektu) | Визначає типи робіт по milestones, призначає старших бригадирів, планує ресурси (двоетапно) |
| **Керівник відділу** (Kierownik Działu) | Роль глядача з повним доступом до всіх проектів (бачить навантаження свого відділу) |
| **Старший бригадир** (Starszy Brygadzista) | Заповнює щоденні звіти (денний підсумок) для своїх відділів |
| **Глядач** (Viewer) | Бачить всі проекти без можливості редагування |
| **Контрагент** (Contractor) | Авторизується в системі, бачить прогрес тільки по своїх проектах (наживо) |

## 3. Бізнес-процес (послідовність дій)

### Крок 1 — Створення проекту
**Хто:** Операційний директор

Заповнює основні дані проекту: контрагент, назва, локація (адреса + GPS), країна, дати, тип проекту, специфічні параметри (напр. для PV — Power MW), призначає менеджера.

### Крок 2 — Фінансові та гарантійні умови
**Хто:** Операційний директор

**Частина А** — Загальні умови контракту: термін платежу, % гарантії, період гарантії, нетто-вартість, VAT, валюта.

**Частина Б** — Milestones (кам'яниці): номер, опис етапу, % від вартості, нетто-сума, % фактурування (як часто можна виставляти часткові фактури).

### Крок 3 — Планування бюджету та розрахунок годин
**Хто:** Операційний директор

**Етап 3.1** — Вибір статей витрат з довідника (зарплати, важка техніка, хостели, матеріали, субпідрядники, транспорт, інвестиції тощо) та встановлення планової суми для кожної.

**Етап 3.2** — Для статті "Зарплати" директор вводить середню годинну ставку → система рахує планову кількість годин на весь проект:
```
planned_hours_total = salary_budget / average_hourly_rate
```

**Етап 3.3** — Після того, як менеджер додасть типи робіт (Крок 4), директор (радячись з менеджером) розподіляє години відсотково між типами робіт. Він може орієнтуватися на статистику з попередніх схожих проектів, але рішення приймає індивідуально (через різницю у ґрунтах, погоді, вимогах контрагента).

### Крок 4 — Визначення зобсягу робіт та призначення бригадирів
**Хто:** Менеджер проекту

**Етап 4.1** — Для кожного milestone додає типи робіт (rodzaje prac): назва (вільний ввід з підказками з попередніх проектів), одиниця виміру (шт/м/м²/%/кг), загальна кількість, планові дати початку/кінця, прив'язка до відділу.

**Вимога:** кожен milestone повинен мати мінімум один тип робіт, і кожен тип робіт прив'язаний до одного відділу.

**Етап 4.2** — Для кожного задіяного на проекті відділу призначає відповідального старшого бригадира (один бригадир може відповідати за кілька відділів).

### Крок 5 — Детальне планування ресурсів
**Хто:** Менеджер проекту

Для кожного типу робіт (НЕ milestone в цілому, як було в старій системі) менеджер планує: кількість працівників, дату початку, дату кінця. План можна редагувати в процесі.

### Крок 6 — Щоденне звітування
**Хто:** Старший бригадир

Дані щодо працівників та годин беруться з **окремого існуючого модуля** (НЕ в цій базі). У нашу систему імпортується тільки **денний підсумок** у форматі:
- Дата
- Проект
- Тип робіт
- Кількість працівників (факт)
- Сума годин (факт)
- Кількість виконаних робіт (факт) — в одиницях виміру з Кроку 4

### Крок 7 — Фінансовий цикл
**Хто:** Фінансовий директор

Планування доходів/витрат, виставлення фактур, прив'язаних до milestones, облік платежів, аналіз рентабельності, cash flow, прострочок.

*(Детальна логіка виставлення фактур та % фактурування буде опрацьована пізніше.)*

## 4. Архітектурні рішення (зафіксовано)

### 4.1 Універсальність типів проектів
- **Загальні поля** — як окремі колонки в таблиці `projects` (назва, контрагент, дати, локація тощо).
- **Специфічні поля під тип проекту** — в колонці `custom_fields JSONB` з валідацією на бекенді.
- Приклад: для PV — `{"power_mw": 5.2}`, для BESS — `{"capacity_mwh": 10, "power_mw": 2.5}`.

### 4.2 Типи проектів
Окрема таблиця-довідник `project_types`. Можна додавати нові типи без зміни коду.

### 4.3 Відділи будови (departments)
**Глобальний довідник** (один `Kafar` на всю систему — старший бригадир з цього відділу може працювати на будь-якому проекті з будь-яким типом).

**Зв'язок "який відділ доступний для якого типу проекту"** — join-таблиця `project_type_departments`.

**На конкретному проекті** — задіяні не обов'язково всі відділи типу проекту; відділ вважається задіяним, якщо до нього прив'язаний хоча б один тип робіт.

### 4.4 Типи робіт (rodzaje prac)
Вільний ввід (`VARCHAR(255)`). Підказки з попередніх проектів — **логіка фронтенду** (запит з унікальних значень попередніх проектів того ж типу), в БД не зберігається довідник.

### 4.5 Статті витрат (cost categories)
Фіксований довідник `cost_categories` з можливістю додавати нові. У проекті вибираємо які статті використовуємо та суму для кожної — таблиця `project_budget_items`.

**Спеціальний прапорець** `is_salary = true` для статті "Зарплати" — з неї йде розрахунок годин.

### 4.6 Розподіл годин по типах робіт
Відсотки розподіляються на **рівні типу робіт** (не на рівні відділу). 

**Алгоритм:**
1. Зарплатний бюджет ÷ середня ставка = загальні години
2. Операційний директор розподіляє загальні години відсотково між усіма типами робіт проекту
3. Зберігаємо `project_id | work_type_id | planned_hours | percentage`

Система **не підраховує статистику автоматично** — директор сам аналізує попередні проекти перед рішенням.

### 4.7 Мультивалютність
Валюта **на рівні проекту** (один проект = одна валюта, вказується в `projects.currency`). 

VAT — вільне значення (не enum), можна вказати будь-яку ставку (0%, 5%, 8%, 23% або інше).

### 4.8 Soft delete
Для критичних сутностей додаємо `deleted_at TIMESTAMP NULL` (soft delete): projects, milestones, work_types, invoices, daily_reports.

### 4.9 Аудит змін
Поля `created_at`, `updated_at`, `created_by`, `updated_by` — для всіх основних таблиць.
Детальний audit log (окрема таблиця `audit_log`) — опціонально, обговорюється пізніше.

### 4.10 Версіонування планів
Поки що **last write wins**. Історію змін можна додати пізніше через audit_log.

### 4.11 Статуси проекту (project lifecycle)

Проект має **життєвий цикл** — послідовність статусів, через які він проходить від створення до архівації. Це критично для коректної роботи звітності, планування ресурсів та cash flow.

**Список статусів:**

| Статус | Значення | Коли використовується |
|---|---|---|
| `draft` | Чернетка | Створено, але ще не доповнено (немає менеджера, milestones, бюджету) |
| `active` | У роботі | Бригади звітують, йде планування ресурсів, прогнозуються доходи |
| `on_hold` | Призупинено | Тимчасова пауза з зовнішньої/внутрішньої причини |
| `completed` | Завершено | Всі milestones закриті, всі роботи виконані |
| `archived` | Архів | Оплати отримані, гарантійний період минув, "холодне" зберігання |
| `cancelled` | Скасовано | Проект скасовано (відмова контрагента, банкрутство, інше) |

**Типові причини `on_hold`:**
- **Зовнішні:** неоплата контрагентом попереднього етапу, затримка дозволів (UDT, оператор мережі), зміни проектної документації, юридичні спори по ділянці
- **Сезонні:** зимова пауза (для PV — проблеми з забиванням паль у промерзлий ґрунт), погодні умови
- **Внутрішні:** затримка постачання (панелі, інвертори, кабелі AC), перекидання бригад на пріоритетніший проект, поломка ключової техніки

**Життєвий цикл (state transitions):**

```
draft ──▶ active ──▶ completed ──▶ archived
            │  ▲
            ▼  │
         on_hold
            │
            ▼
        cancelled
```

Дозволені переходи:
- `draft` → `active` або `cancelled`
- `active` → `on_hold`, `completed` або `cancelled`
- `on_hold` → `active` або `cancelled`
- `completed` → `archived`
- `archived` → (кінець; тільки перегляд)
- `cancelled` → (кінець; тільки перегляд)

**Вплив статусу на бізнес-логіку:**

| Статус | Щоденні звіти | Планування ресурсів | Cash flow | Видно бригадирам як активний |
|---|---|---|---|---|
| `draft` | — | — | — | — |
| `active` | ✅ обов'язково | ✅ враховується | ✅ активний | ✅ |
| `on_hold` | ❌ призупинено | ❌ не враховується | ⏸ заморожено | ⚠️ з міткою паузи |
| `completed` | — | — | ✅ фінальні платежі | — |
| `archived` | — | — | — | — |
| `cancelled` | — | — | — | — |

**Вимоги до реалізації в БД:**

1. **Колонка `projects.status`** — ENUM або FK на `project_statuses` (довідник). Рекомендовано ENUM для простоти.

2. **Обов'язкові поля для `on_hold`:**
   - `hold_reason TEXT NOT NULL` (обов'язкова причина при переході в `on_hold`)
   - `hold_started_at TIMESTAMPTZ NOT NULL`
   - `expected_resume_date DATE NULL` (необов'язкове, якщо відомо)
   
   Валідація: при переході в `on_hold` ці поля мають бути заповнені. При виході з `on_hold` в `active` — очистити (або залишити для історії? — див. пункт 3).

3. **Таблиця `project_status_history`** — окрема таблиця історії змін статусу:
   - `id`
   - `project_id` (FK)
   - `old_status`
   - `new_status`
   - `reason TEXT` (обов'язково для `on_hold` і `cancelled`)
   - `changed_by` (FK на users)
   - `changed_at TIMESTAMPTZ`
   
   Це дає повну історію, навіть якщо проект кілька разів призупинявся.

4. **CHECK constraint або trigger** на `projects` — валідація дозволених переходів (не можна з `draft` одразу в `completed`, не можна з `archived` назад в `active` тощо).

5. **Партіал-індекси** для оптимізації запитів по активних проектах:
   ```sql
   CREATE INDEX idx_projects_active ON projects(id) 
   WHERE status = 'active' AND deleted_at IS NULL;
   ```

## 5. Перелік сутностей (entities) — попередній список

### Довідники (dictionaries / lookup tables)
1. `users` — користувачі системи
2. `roles` — ролі (operational_director, financial_director, project_manager, department_head, foreman, viewer, contractor)
3. `user_roles` — M2M зв'язок користувачів та ролей
4. `contractors` — контрагенти (замовники)
5. `project_types` — типи проектів (PV, BESS, майбутні)
6. `departments` — відділи будови (Kafar, Montaż, Elektryka, Maszyny budowlane, Kable AC)
7. `project_type_departments` — M2M: які відділи доступні для якого типу проекту
8. `cost_categories` — статті витрат (з прапорцем `is_salary`)
9. `currencies` — довідник валют (PLN, EUR, USD тощо) — опціонально, можна ISO-код зберігати напряму

### Ядро проекту
10. `projects` — проекти (з колонкою `status`, `hold_reason`, `hold_started_at`, `expected_resume_date`)
11. `project_status_history` — історія змін статусу проекту (хто, коли, з якого на який, причина)
12. `project_managers` — призначення менеджерів (історично, якщо потрібно замінити менеджера)
13. `milestones` — кам'яниці (етапи проекту)
14. `work_types` — типи робіт (прив'язані до milestone + department)
15. `project_foremen` — призначення старших бригадирів по відділах у конкретному проекті

### Бюджет та години
16. `project_budget_items` — статті витрат проекту з сумами
17. `project_hours_plan` — загальна кількість годин (середня ставка + сума = розрахунок)
18. `work_type_hours_distribution` — розподіл годин по типах робіт

### Планування ресурсів
19. `resource_plans` — детальне планування працівників на типи робіт (дата початку, дата кінця, кількість)

### Факт (денні звіти)
20. `daily_reports` — денні підсумки (імпортовані з існуючого модуля або вводяться вручну)

### Фінанси
21. `invoices` — фактури
22. `invoice_payments` — платежі по фактурах (для підтримки часткових платежів)
23. `planned_revenues` — планові доходи (прив'язані до milestones)
24. `planned_expenses` — планові витрати (прив'язані до cost_categories)

### Доступи
25. `contractor_project_access` — до яких проектів має доступ користувач-контрагент

## 6. Ключові вимоги до схеми

### 6.1 Обов'язкові для всіх основних таблиць
- `id` — UUID або BIGSERIAL (на вибір розробника)
- `created_at TIMESTAMPTZ DEFAULT NOW()`
- `updated_at TIMESTAMPTZ DEFAULT NOW()`
- `created_by` — FK на `users.id`
- `updated_by` — FK на `users.id`

### 6.2 Обов'язкові для soft-delete таблиць
- `deleted_at TIMESTAMPTZ NULL`
- Фільтрація `WHERE deleted_at IS NULL` у всіх запитах на читання

### 6.3 Грошові поля
- `NUMERIC(15,2)` для сум
- `NUMERIC(5,2)` для відсотків (щоб вміщало 100.00 + знаки)
- Валюта на рівні проекту (`projects.currency CHAR(3)` — ISO 4217)

### 6.4 Обмеження (constraints)
- Унікальність назв проектів в межах контрагента
- `CHECK` на відсотки (0–100)
- `CHECK` на дати (`date_to >= date_from`)
- `CHECK` на суму milestones (= 100% контракту) — через trigger або на рівні валідації
- FK з `ON DELETE RESTRICT` для критичних зв'язків (не дозволяти видалити проект, якщо є фактури)

### 6.5 Індекси
- На всі FK
- На `projects.contractor_id`, `projects.status`, `projects.project_type_id`
- На `daily_reports.report_date` + `project_id` (композитний)
- На `deleted_at IS NULL` (partial index) для таблиць з soft-delete

## 8. Важливі нюанси для розробника

1. **Денний звіт (daily_reports) — спрощений.** Детальні per-працівник дані в іншому модулі. Сюди пишемо агрегат: дата + проект + тип робіт + кількість працівників + години + обсяг.

2. **Бюджет і години — ланцюжок залежностей:**
   - `project_budget_items` (стаття "Зарплати" з сумою)
   - → `project_hours_plan` (сума / ставка = години)
   - → `work_type_hours_distribution` (%-и по типах робіт)
   
   Розподіл годин стає можливим **тільки після** того, як менеджер додасть типи робіт (Крок 4). Система повинна обробляти цей флоу.

3. **Milestone → Work Type → Department.** Кожен `work_type` прив'язаний до одного `milestone` (FK) та одного `department` (FK). Відділ вважається "задіяним" у проекті, якщо є хоч один work_type, прив'язаний до нього.

4. **Призначення бригадира.** На рівні проект + відділ (`project_foremen`). Один бригадир може бути призначений на кілька відділів одного проекту.

5. **Контрагент-юзер бачить тільки свої проекти** — через `contractor_project_access` або через `contractors.user_id` (якщо 1:1). Потрібна row-level security або фільтрація на рівні бекенду.

6. **VAT — вільне значення.** Не enum, не довідник. `NUMERIC(5,2)` з CHECK (0–100).

7. **Валюта — на рівні проекту.** Всі суми в цьому проекті (бюджет, фактури, платежі) в одній валюті.

8. **Soft delete критичний** для проектів, фактур, milestones, звітів — будівельна документація часто потрібна для аудиту.

9. **Статуси проекту — НЕ декоративні.** Статус напряму впливає на бізнес-логіку: `on_hold` виключає проект зі щоденного звітування та з планування ресурсів, `archived` — тільки read-only. Переходи між статусами мають валідуватися на рівні БД (CHECK / trigger) або бекенду. Для `on_hold` обов'язково зберігати причину (`hold_reason`) та дату початку паузи. Вся історія змін — в окремій таблиці `project_status_history`.
